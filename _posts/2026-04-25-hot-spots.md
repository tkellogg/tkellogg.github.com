How do you scale out AI use throughout a software engineering org? Do the PM & Engineer roles merge?
I think it's worth stepping back and looking at it through a familiar lens — distributed systems.

I hope you're familiar with the concept of partitioning. The idea is, if a table is receiving too much
traffic, you can split the table into 2 parts (partitions) and each table only needs to handle half the traffic.
Then you relocate those partitions onto different computers, and voila! Scale.

![animation demonstrating how hot spots in a distributed database can look like hot spots in an organization][hspot]

The [web scale][mongo] era was dominated by partitioning. If you can figure out how to partition any kind
of load whatsoever, then you can figure out scale. [Shared nothing][nothing] emerged — any kind of shared 
state is a hot spot liability and must be removed. 

It started with databases but it infected the entire software stack. Load balancers, web services, control plane
/ data plane split, deployments, etc. If you can identify the shared state, you can eliminate them. You 
can scale. 


## "Vibe coding doesn't scale"
[People are noticing][vibes] that vibe coding causes problems. Throw AI tools at a team, suddenly the cost
of software is near zero. Thousands of lines of code fall out effortlessly. The new problem: conflicting changes.

Is this... a hot spot?

What if we viewed a code base as if it were a distributed database. Instead of _traffic_, let's look at _change_.
Everyone on the team is making changes all at once with their agents. An agent can write 10k lines of code in the
time it takes to have a meeting about retries. Claude Code can autonomously debug and fix a gnarly bug while you
get coffee.

But a merge conflict? Everything stops to wait on the humans.

![](/images/vibe-numbers.png)

What if we introduced **shared nothing** architecture to this? We could view _code changes_ the same way
we view traffic flow in distributed systems.


## Partitioning the code base
First of all, we know that vibe coding a **prototype is easy**, but working on an established code base is hard.
Why not lean into that?

PARTITION THE CODE

It seems, then, that a well-designed code base should be small and focused. So if you want to build a big
product, it should ideally be composed of lots of tiny components that can each be rewritten on a whim.


## Not just the code
Do we need both PM & engineers? I think when we start viewing change as a traffic pattern, the PM->Engineer
pipeline only makes sense when you think of those two as incapable of doing each other's jobs. But LLMs are
general intelligences. An agent might exude a vibe, but they're still quite capable of doing a wide range of
tasks.

Worse, we already know from the Product->Engineeer->QA pipeline that each of those jumps introduces the
potential for miscommunication. Plus, the hand-offs are costly.

I think the most efficient combo might be to treat a human + agent pair as a unit and assign end-to-end
responsibility. Product, engineering & QA.

## Conway's Law cuts both ways

[Conway][conway] told us code structure mirrors org structure. We invoke this every time we want to justify
a reorg, but we forget it goes the other way too: partition the code and the org HAS to follow.

The PM → Eng → QA pipeline is a Conway-shaped artifact of monolithic codebases. One thing was being built,
so one pipeline funneled decisions toward it. Partition the code into vertical slices and that pipeline is
gravity-defying. It dissolves on its own.

So it's not that AI killed the PM. It's that **shared-nothing code makes PM → Eng → QA handoffs unbillable.**

## But what doesn't partition?

Here's where the metaphor gets honest. We partitioned databases. We partitioned load balancers. We partitioned
services. We never partitioned NTP. We never partitioned DNS roots. We never partitioned the IP layer.

Some things are irreducibly shared. The base of the stack stays singular while everything above it scales out.

Org equivalent? Auth. Billing. Customer view. Regulatory posture. Identity. Trust. Incident response.

If every vertical slice mints its own auth tokens, you have a security incident in 18 months. If every slice
owns its own billing, your CFO can't close the books. If every slice owns its own observability, on-call can't
triage anything.

The shared-nothing pitch always undercounts the shared core. That's how microservices became distributed
monoliths the first time around — the partitioning was real, the platform team that PAID for the partitioning
was a surprise.

## The platform-team-shaped hole

So the human+agent unit owns end-to-end... what exactly? Their slice. Not auth. Not billing. Not observability.
Not security review. Not incident response. Not regulatory compliance. Not the customer view.

Someone owns those. In distributed systems we built SRE. We built platform engineering. We built sidecars and
service meshes. The partitioning came WITH a new capability layer — observability, chaos engineering, consistency
primitives — that didn't exist before.

Org-as-distributed-system is going to need its own version. Something like:

- **Variety budgeting** — who decides the scope of what each slice can change?
- **Change-impact observability** — when slice A changes, what else does it touch?
- **Slice-coupling metrics** — which slices are too entangled to be one-person units?

I don't know what these look like yet.

## We have a pattern, not a theory

That's the honest landing. CAP took 30 years after Codd. Raft took a decade after Paxos. The sidecar pattern
emerged after we'd already built three generations of distributed monoliths.

We're somewhere around 2008-MongoDB on this one. Lots of confidence. Not a lot of theory.

A human+agent unit might handle a vertical slice. But two units sharing a customer view will rediscover the
merge conflict — it just moved up a level, from code to product. You can't partition that away.

The end state probably isn't "no PMs." It's "PM, Eng, and QA collapse INTO the slice owner, AND a smaller
residue of cross-cutting concerns becomes its own discipline that didn't have a name yet." Like SRE in 2003.

The interesting question isn't "do PMs survive." It's "what's the SRE-equivalent for AI-augmented orgs, and
who's paying their salary?"
