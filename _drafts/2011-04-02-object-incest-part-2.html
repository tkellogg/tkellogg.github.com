In my last post I wrote about an anti-pattern called object incest. After talking it through with some coworkers I realized that it wasn't completely clear what this is and why it's a bad thing. The biggest reason I want to readdress this is because it happens a lot on ASP.NET pages, and I have to work with these pages and the weight of the debt caused by these pages is too much to get any real work done.

Object incest, to reiterate, happens when object A directly depends on object B and vice versa. These don't have to be objects. An ASP.NET page is an object and so is a control. In fact, an object can really be any entity, like a POCO, page, service, or database object.
